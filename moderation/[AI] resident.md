Не обязательно. Опция `-O resident` для `ssh-keygen -t ed25519-sk` нужна **только если ты хочешь “резидентный” (discoverable) ключ**, то есть чтобы *сам ключ хранился в YubiKey* и его можно было “поднять” на другом компьютере без файла `~/.ssh/id_ed25519_sk`.

### Что происходит без `-O resident` (по умолчанию)
- На YubiKey хранится FIDO2-ключ (секрет), но на диске создаётся файл `~/.ssh/id_ed25519_sk`, который содержит **ссылку/дескриптор (key handle)** на этот ключ.
- Чтобы аутентифицироваться, обычно нужен **и YubiKey, и этот файл**.
- Плюсы: проще, меньше возни, не упираешься в ограничения resident-слотов.
- Для обычного сценария “одна рабочая машина → git push” этого достаточно.

### Что даёт `-O resident`
- Ключ становится **discoverable/resident**: YubiKey может “сам” предъявить его без заранее сохранённого key handle файла.
- На новой машине можно импортировать “заглушки/публичные части” командой вроде:
  ```bash
  ssh-keygen -K
  ```
  (считает resident-ключи с токена и запишет их в `~/.ssh/`)

- Минусы/ограничения:
  - обычно **требует PIN** (и часто правильно делать вместе с `-O verify-required`);
  - на устройстве **ограниченное количество** resident-ключей (зависит от модели/прошивки);
  - управление сложнее (при сбросе FIDO2 или потере PIN можно потерять доступ);
  - если у тебя уже добавлен SSH ключ в GitHub/GitLab, при переходе на resident ты фактически создаёшь **новую пару ключей** → нужно будет добавить новый `.pub` в хостинг.

### Рекомендация
- Если тебе нужно просто “git push с подтверждением касанием” на одной машине: **`-O resident` не нужен**.
- Если ты хочешь “подключил YubiKey к любому ПК и сразу могу ssh/git”, без переноса файлов из `~/.ssh`: **тогда да, resident уместен**.

### Как правильно генерировать resident-ключ (типовой вариант)
```bash
ssh-keygen -t ed25519-sk -O resident -O verify-required -C "yubikey-ssh" -f ~/.ssh/id_ed25519_sk
```
